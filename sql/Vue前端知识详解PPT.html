⚡ Vue 3 核心机制：响应式系统（Reactivity）

ref() — 基本类型响应式
// 包装基本值：string/number/boolean
const count = ref(0);
const name = ref('张三');
const loading = ref(false);

// 也能包装对象/数组
const user = ref<User | null>(null);
const list = ref<Item[]>([]);

// JS 中操作需要 .value
count.value++;                // 模板里自动解包
user.value = { name: '李四' };
list.value = newData;         // 替换整个数组

// 模板中不用 .value
// <div v-if="loading">加载中...</div>
// <span>{{ user?.name }}</span>
reactive() — 对象响应式
// 包装对象，不需要 .value
const form = reactive({
  username: '',
  password: ''
});

// 直接读写属性
form.username = 'admin';
form.password = '123456';

// 也可以包装数组
const items = reactive([]);
items.push({ id: 1 });

// ⚠️ 不可解构，会丢失响应式！
// const { username } = form;  ← 错误

// 适合：表单数据、配置对象
const pagination = reactive({
  currentPage: 1,
  pageSize: 50,
  total: 0

computed() — 计算属性
// 从现有数据派生新值，有缓存！
const firstName = ref('张');
const lastName = ref('三');

const fullName = computed(() =>
  firstName.value + ' ' + lastName.value
);

// 项目中的典型用法
// 1. 只读判断

  props.isReadOnly ?? false
);
// 2. 动态请求头
const uploadHeaders = computed(() =>
  ({ Authorization: auth.authHeader })

// 3. 权限判断

📊 ref vs reactive — 什么时候用哪个？

场景推荐原因
基本类型（字符串/数字/布尔）refreactive 不支持基本类型
表单数据对象reactive不需要 .value，和模板绑定简洁
API 返回的对象ref<T | null>可以整体替换为 null
数组（列表数据）ref<T[]>方便整体替换 .value = newData
分页配置reactive多属性分组，语义清晰
组件实例引用ref模板 ref 唯一支持
跨组件共享状态ref（在 Pinia 中）Pinia setup 风格推荐 ref
⚠️ reactive 的限制
① 不能整体替换：form = newObj 会丢失响应式
② 不能解构：const { a } = obj 失去响应式
③ 只接受对象/数组：不能直接放基本值
项目中的实际用法
// 单个加载状态 → ref
const loading = ref(false);
const errorMsg = ref('');

// API 返回数据 → ref<类型 | null>
const user = ref<UserInfo | null>(null);
const tableData = ref<ComponentItem[]>([]);

// 表单数据 → reactive
const loginForm = reactive({
  username: '', password: ''
});

// 验证规则 → reactive（对象）
const rules = reactive<FormRules>({...});

// 组件引用 → ref
const formRef = ref<FormInstance>();
formRef.value?.validate(...);
💡 一句话总结
大部分场景用 ref，表单/配置类对象用 reactive。
不确定就用 ref，它最通用。

👀 watch 监听 & 生命周期钩子

watch — 监听数据变化
// 1. 监听 ref
watch(count, (newVal, oldVal) => {
  console.log(`从 ${oldVal} 变为 ${newVal}`)
})

// 2. 监听 reactive 属性（getter 函数）
watch(
  () => form.username,
  (newName) => { /* 用户输入时触发 */ }
)

// 3. 监听多个源
watch(
  [count, () => form.username],
  ([c, name]) => { ... }
)

// 4. 深度监听（对象内部变化）
watch(form, (val) => {
  // form 任何字段变化都触发
}, { deep: true })

// 5. 立即执行（组件挂载时就跑一次）
watch(source, fn, { immediate: true })
项目中的 watch 实战（App.vue）
// 监听路由变化，刷新认证状态

  () => route.path,

    if (newPath !== '/login') {

子传父 — Emits
// === 子组件 ===
const emit = defineEmits<{
  // 事件名: (参数) => 返回值
  (e: 'close', needRefresh: boolean): void
}>();

// 触发事件，携带参数
emit('close', true);

// === 父组件 ===

  @close="handleDialogClose"

  if (refresh) fetchData();
};

Props 向下（父→子），只能读不能改
Emits 向上（子→父），通过事件通知
兄弟组件通信 → 共享父组件状态 或 Pinia
defineProps声明组件接收哪些参数
defineEmits声明组件会触发哪些事件
withDefaults给 Props 设置默认值

beforeEach 全局前置守卫
// 每次路由切换前都会执行
router.beforeEach((to, from, next) => {
  // ① 已登录访问登录页 → 跳首页
  if (to.path === '/login' && isLoggedIn()) {
    next('/'); return;
  }
  // ② 未登录访问需认证页 → 跳登录
  if (to.meta.requiresAuth && !isLoggedIn()) {
    next({ path: '/login',
              query: { redirect: to.fullPath } });
    return;
  }
  // ③ 非管理员访问管理员页 → 跳首页
  if (to.meta.requiresAdmin && !isAdmin()) {
    next('/'); return;
  }
  // ④ 检查页面权限（非管理员）
  const perm = to.meta.pagePermission;
  if (perm && !isAdmin()) {
    if (!getPerms().includes(perm)) {
      next('/'); return;
    }
  }
  next(); // 放行
});
afterEach — 设置页面标题
router.afterEach((to) => {
  document.title =
    `${to.meta.title} - WKB`;
});
路由元信息 meta 的 3 种角色
requiresAuth: true需要登录未登录→跳登录页
requiresAdmin: true仅管理员非 ADMIN→跳首页
pagePermission: 'page4'需要页面权限无权限→跳首页
💡 next() 三种调用方式
next() — 放行，进入目标路由
next('/login') — 重定向到登录
next(false) — 取消导航

📡 Axios 封装 — HTTP 层的统一管理

请求拦截器 — 自动注入 Token
// src/api/request.ts
const service = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 15000,  // 15秒超时
  headers: { 'Content-Type': 'application/json' }
});

// 请求拦截器：每次请求前自动执行
service.interceptors.request.use(
  (config) => {
    const token = getToken();
    if (token) {
      // 自动在 HTTP 头中添加 Authorization
      config.headers.set(
        'Authorization',
        `Bearer ${token}`
      );
    }
    return config;
  }
);
响应拦截器 — 自动处理错误
service.interceptors.response.use(
  // 成功：自动解包 response.data
  (response) => response.data,

  async (error) => {
    const status = error.response?.status;

    if (status === 401) {
      // Token 过期 → 自动尝试刷新
      if (!isRefreshing) {
        await refreshToken();
        // 刷新成功 → 重试原请求
        return service(error.config);
      }
      // 刷新失败 → 强制重新登录
      router.push('/login');
    }

    if (status === 403) {
      // 无权限提示
    }
    return Promise.reject(error);
  }
);
Token 刷新队列（并发请求处理）
// 问题：同时发出 3 个请求，都 401
// 方案：只在第一个触发刷新，其余排队
let isRefreshing = false;
let queue = [];

if (isRefreshing) {
  // 先进入等待队列
  return new Promise(resolve => {
    queue.push(token =>
      resolve(service(request)));
  });
}
isRefreshing = true;
// 刷完通知所有排队的请求
queue.forEach(cb => cb(newToken));

🍍 Pinia — 全局状态管理（替代 Vuex）

为什么需要状态管理？
每个 .vue 组件都有自己独立的数据。但 Token、用户信息、权限列表这些数据需要被多个组件共享。
如果用 Props 一层层传递太麻烦 → 用 Pinia 做全局数据仓库，任何组件都能直接读写。 
// 组件 A（App.vue）
const auth = useAuthStore();
auth.hasPerm('page4');   // 读权限

// 组件 B（Login.vue）
await auth.login(...);         // 写状态

// 组件 C（request.ts）
const { authHeader } = useAuthStore();

// ⭐ 三个地方访问的是同一份数据！
//    B 登录 → A 和 C 自动感知变化
Setup Store 语法（推荐）
export const useAuthStore =
  defineStore('auth', () => {

  // ===== 状态（state） =====
  const token = ref(getToken());
  const isLoggedIn = ref(false);

  // ===== 计算（getters） =====
  const authHeader = computed(() =>
    token.value
      ? `Bearer ${token.value}`
      : null
  );

  // ===== 方法（actions） =====
  async function login(params) {
    const res = await loginApi(params);
    token.value = res.data.token;
    isLoggedIn.value = true;
  }

  return { token, isLoggedIn, authHeader, login };
});

① 更简洁的 API，没有 mutations
② 完美支持 TypeScript
③ Setup Store 风格与 Composition API 一致

双重存储策略
// Pinia 状态只在内存中，刷新页面就没了
// 所以需要 localStorage 做持久化

// 登录时同时写入两处
const login = async (params) => {
  const res = await loginApi(params);

  // ① 写 Pinia（内存，响应式）
  token.value = res.data.token;
  isLoggedIn.value = true;

  // ② 写 localStorage（磁盘，持久化）
  localStorage.setItem('token',
    res.data.token);
  localStorage.setItem('tokenType',
    res.data.tokenType);
  localStorage.setItem('expiresAt',
    String(Date.now() +
    res.data.expiresIn * 1000));
};

📝 Vue 3 模板语法 — 最常用的 6 个指令

① v-model — 双向绑定
<!-- 输入框 ↔ 数据 自动同步 -->
<input v-model="form.name" />

// 等价于
<input
  :value="form.name"
  @input="form.name = $event"
/>

// 修饰符
v-model.trim      // 自动去空格
v-model.number    // 自动转数字
② v-if / v-show
<!-- v-if: 条件为真才渲染 -->
<div v-if="isLoggedIn">
  欢迎回来
</div>

<!-- v-else -->
<div v-else>请先登录</div>

<!-- v-show: display:none 切换 -->
<div v-show="visible">可见</div>

// v-if: 真正的插入/移除 DOM
// v-show: 只是 CSS 隐藏，DOM 还在
// 频繁切换用 v-show，条件很少变用 v-if
③ v-for — 列表渲染
<!-- 数组遍历 -->
<div
  v-for="item in list"
  :key="item.id"
>
  {{ item.name }} - {{ item.status }}
</div>

<!-- 带索引 -->
<div v-for="(item, index) in list"
  :key="index"
>

<!-- ⚠️ :key 必须唯一！
     不要用 index 做 key（列表会变） -->
④ @click / @change — 事件
// @事件名="处理函数"
@click="handleSubmit"
@change="handleCompanyChange"
@keyup.enter="handleLogin"

// 带参数
@click="handleEdit(row)"
@click="count++"
⑤ :bind — 动态属性
// :属性名="变量/表达式"
:disabled="!isEditable"
:loading="loading"
:class="{ active: isActive }"
:style="{ color: textColor }"
⑥ {{ }} — 插值
// 输出变量或表达式
{{ message }}
{{ user.name }}
{{ count + 1 }}
{{ isAdmin ? '管理员' : '用户' }}
{{ loading ? '提交中' : '提交' }}

🔑 项目核心模式：从点击按钮到页面刷新

一个完整的增删改查流程
// ① 用户点击「查询」按钮
<button @click="fetchData">查询</button>

// ② 触发异步加载函数
const tableLoading = ref(false);
const tableData = ref<Item[]>([]);

async function fetchData() {
  tableLoading.value = true;  // 开启加载
  try {
    const res = await getList({
      page: pagination.currentPage,
      pageSize: pagination.pageSize,
      ...searchForm
    });
    tableData.value = res.data.records;
    pagination.total = res.data.total;
  } catch (e) {
    ElMessage.error('加载失败');
  } finally {
    tableLoading.value = false; // 关闭加载
  }
}
流程中的关键模式
loading 状态ref(true/false)，控制骨架屏和按钮
try/catch捕获错误，不阻塞 UI
finally无论成败都关闭 loading
.value 赋值更新数据后，模板自动刷新
分页联动切换页码 → 重新 fetchData()
删除确认 + ElMessageBox
// 删除前必须确认
async function handleDelete(row) {
  try {
    await ElMessageBox.confirm(
      '确定要删除吗？',
      '提示',
      { type: 'warning' }
    );
    await deleteApi(row.id);
    ElMessage.success('删除成功');
    fetchData();  // 刷新列表
  } catch (e) {
    if (e !== 'cancel') {
      ElMessage.error('删除失败');
    }
  }
}

🔗 权限与菜单联动 — 前后端双控

菜单按权限显示（App.vue）
// 侧边栏菜单项根据权限动态显示
<el-menu-item
  v-if="auth.hasPerm('page1')"
  index="/page1"
>
  页面一
</el-menu-item>

<el-menu-item
  v-if="auth.hasPerm('page4')"
  index="/page4"
>
  工装申请管理
</el-menu-item>

// 管理员专属菜单
<el-menu-item
  v-if="auth.isAdminUser"
  index="/users"
>
  用户管理
</el-menu-item>
权限流转链路
① 登录 → 后端返回 token + 角色 + 权限列表
② 存储 → localStorage + Pinia 双重持久
③ 菜单 → v-if="auth.hasPerm('page1')" 控制显示
④ 路由 → beforeEach 守卫二次校验
⑤ 页面 → isReadOnly / isAdmin 控制按钮状态
⑥ 请求 → 后端 Spring Security 最终校验
⚠️ 前端权限只是"门面"
前端做权限控制是为了用户体验（隐藏无权的菜单/按钮）
真正的安全保障在后端 Spring Security + JWT 校验
前端永远不可信！

🧠 从 MiniUI 到 Vue 3 — 思维模式转变

❌ MiniUI 思维方式
🔴 页面是 静态 HTML，JS 去操作 DOM
🔴 数据藏在 DOM 里，取值靠 getValue()
🔴 更新界面靠 setValue() / setEnabled()
🔴 页面跳转 = window.location.href（全量刷新）
🔴 表单提交靠 手动拼装参数
🔴 列表刷新 = grid.load()（重渲染整表）
🔴 没有"状态"概念，DOM 就是状态
✅ Vue 3 思维方式
🟢 页面是 数据的投影，数据变则界面变
🟢 数据在 JS 变量中，界面靠 v-model 自动同步
🟢 更新界面 = ref.value = 新值（不用管 DOM）
🟢 页面跳转 = router.push()（无刷新 SPA）
🟢 表单提交 = 直接 { ...form } 拿到数据
🟢 列表刷新 = list.value = newData（自动更新）
🟢 一切皆状态，界面只是状态的渲染结果
核心转变：从「操作 DOM」到「管理数据」
DOM 不再是你要关心的东西，框架帮你处理好了。你只需要思考数据是什么样的、数据怎么变。
